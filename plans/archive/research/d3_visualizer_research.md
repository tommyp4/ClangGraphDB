## 1. Executive Summary

The objective is to create a single-page web application using D3.js to visualize, navigate, and search the Code Property Graph (CPG) generated by the `graphdb` tool. This visualizer will enable users to explore system architecture, trace dependencies, discover semantic similarities via vector search, and plan refactoring efforts (e.g., Strangler Fig pattern) interactively.

## 2. System Architecture Mapping (Current vs. Future State)

### Current State
- **Backend/Ingestion:** Go-based CLI (`cmd/graphdb`) that parses codebases across multiple languages (C#, Java, C++, TypeScript, SQL, etc.), generates a structural graph, and imports it into Neo4j.
- **Enrichment:** Go pipeline that clusters nodes semantically, propagates "contamination" layers (UI, DB, External I/O), maps Git history, and links tests to production code.
- **Query Engine:** The `graphdb query` command interacts with Neo4j and Vertex AI (for embeddings) to return JSON results for structural and semantic queries.
- **Database:** Neo4j (local container or managed).

### Target Architecture for D3 Visualizer
To support a web-based D3 visualization, we need to introduce a lightweight API server (or adapt the existing Go codebase to serve HTTP requests) to act as a bridge between the browser and Neo4j/Vertex AI.

1.  **Frontend (Web Browser):**
    *   **Framework:** Vanilla JS or a lightweight framework (React/Vue/Svelte) for state management, hosting the D3.js canvas.
    *   **Visualization Engine:** D3.js (Force-directed graphs for general exploration, Tree/Hierarchical layouts for dependency paths and call stacks).
    *   **Features:** Search bar, node detail panel, filtering toggles (exclude tests, isolate UI contamination), refactoring simulation mode.
2.  **API Gateway / Backend Service (Go):**
    *   The existing `graphdb` query logic (`internal/query/*`) should be exposed via REST or GraphQL endpoints instead of just CLI output.
    *   Endpoints needed:
        *   `GET /api/search?q={query}` -> Maps to `search-similar` or `search-features`.
        *   `GET /api/node/{id}/neighbors?depth={d}` -> Maps to `neighbors` or `hybrid-context`.
        *   `GET /api/node/{id}/impact` -> Maps to `impact`.
        *   `POST /api/simulate/what-if` -> Maps to `what-if` (passing targets to sever/isolate).
        *   `GET /api/system/globals` -> Maps to `globals`.
3.  **Data Layer:**
    *   **Neo4j:** Stores the nodes (Functions, Classes, Files) and edges (CALLS, DECLARES, DEPENDS_ON).
    *   **Vector Search:** Vertex AI or Neo4j Vector Index for embedding-based similarity searches.

## 3. Interaction Model & D3 Visualizations

### A. Initial View & Semantic Search
- **Action:** User enters a natural language query (e.g., "HTTP server API" or "Payment processing logic").
- **Backend:** Calls `search-similar` or `search-features`. Returns top N nodes.
- **D3 View:** A list or a scattered layout of independent nodes. Clicking a node centers the graph and transitions to the Neighborhood View.

### B. Node Navigation (Neighborhood View)
- **Action:** User double-clicks a node to expand its dependencies.
- **Backend:** Calls `neighbors` or `hybrid-context`.
- **D3 View:** A **Force-Directed Graph** expands.
  - **Nodes:** Styled by type (e.g., Classes are squares, Functions are circles). Color-coded by contamination layer (Red = UI, Blue = DB, Green = Core Logic).
  - **Edges:** Directed arrows showing call hierarchy.

### C. Refactoring & Risk Analysis (Impact View)
- **Action:** User right-clicks a node and selects "Calculate Impact" or "Simulate Strangler Fig".
- **Backend:** Calls `impact` or `what-if`.
- **D3 View:** A **Hierarchical Tree or Collapsible Indented Tree** showing the "blast radius" upstream (who calls this) and downstream (what this calls). Nodes that become "orphaned" or edges that are "severed" are highlighted in warning colors (orange/red).

## 4. Recommendations for Architect

1.  **Expose an HTTP API:** The immediate technical prerequisite is wrapping the `internal/query` Go packages in an HTTP server (e.g., using `net/http` or Fiber/Gin). The current CLI outputs JSON, which makes this transition relatively straightforward.
2.  **Pagination and Graph Pruning:** D3 force-directed graphs become sluggish past 500-1000 nodes. The API must enforce strict depth limits (`-depth` parameter) and provide summary nodes (e.g., collapsing an entire namespace into a single module node until expanded).
3.  **Graph Data Format (JGF):** Standardize the API output to a format easily consumable by D3, such as JSON Graph Format (JGF), defining `nodes` and `links` arrays explicitly. The current CLI JSON output may need transformation.
4.  **Fix Neo4j Authentication:** The underlying issue preventing queries during this research must be resolved. The Neo4j container startup script (`neo4j-manager/scripts/start_neo4j_container.sh`) or the Go application's `.env` loading mechanism seems to have a mismatch regarding the default `neo4j/password` authentication. Ensure the Docker container explicitly sets the password without requiring a password change on first login (`NEO4J_AUTH=neo4j/password`).
